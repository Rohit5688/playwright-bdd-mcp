`ContextManager` is a TestForge internal service responsible for maintaining the rolling session context. It ensures every tool call has access to the most relevant, token-efficient view of the project — without requiring re-analysis on every turn.

:::note[Internal Service]
`ContextManager` is not a directly-callable MCP tool. It operates automatically as part of the `generate_gherkin_pom_test_suite` and `inspect_page_dom` pipelines. This page documents its behavior so you can understand and predict context initialization costs.
:::

---

## 🏗️ Core Responsibilities

<CardGrid>
  <Card title="DOM History" icon="document">
    Tracks the last DOM snapshot per page URL. Limits context to **1 snapshot per page** to prevent token bloat. Subsequent calls to the same URL reuse the cached snapshot.
  </Card>
  <Card title="Navigation Graph" icon="random">
    Maintains the live navigation graph built by `discover_app_flow`. Exposes it to the generation pipeline so Page Objects reference real routes.
  </Card>
  <Card title="Structural Map" icon="puzzle">
    Holds the AST-analyzed map of existing Page Objects and step definitions — the "Warm Brain" that prevents re-scanning on every generation call.
  </Card>
  <Card title="Token Counters" icon="approve-check">
    Tracks per-tool token estimates for the session. Surfaced via `get_token_budget`.
  </Card>
</CardGrid>

## 📐 Context Initialization Cost

When a new session begins (or after a full context reset), `ContextManager` performs a **warm-up scan**. This is a one-time cost per session:

| Warm-Up Step | Trigger | Token Cost |
| :--- | :--- | :--- |
| Structural map build | First call to any generation tool | Low (~200 tokens via Turbo Mode) |
| DOM capture | First `inspect_page_dom` call per URL | Medium (~400–800 tokens) |
| Navigation graph load | First `discover_app_flow` result | Low (reads cached JSON) |

After warm-up, subsequent calls in the same session skip these steps entirely.

---

## 🔁 Context Window Management

`ContextManager` enforces the **1-scan-per-page rule**: for any given URL, only the most recent DOM snapshot is kept in the rolling context. Older snapshots are evicted automatically.

This prevents the common failure mode where repeated inspections of the same page accumulate 5–10 stale snapshots and exhaust the token budget.

**Manually reset context** when switching tasks:
> *"Start a new TestForge context — I'm moving to the checkout flow now"*

---

## 🛡️ Security Boundaries

- All file paths are resolved against `projectRoot`. Traversal via `..` is rejected.
- `.env` values and `users.*.json` content are masked in all telemetry logs.
- DOM snapshots stored in memory are never written to disk.
