## 📦 How the Sandbox Works
In traditional MCP servers, the server sends raw data (like a massive JSON file) to the AI. In TestForge, the AI sends a **Script** to the server. The data stays local; only the **Answer** is sent back.

<Steps>

1.  **AI Analysis Intent**  
    The AI agent realizes it needs to find a specific pattern in 50 files.
2.  **Script Generation**  
    Instead of reading 50 files, the AI writes a search script using our optimized API.
3.  **Local Execution**  
    The TestForge server runs the script in a secure V8 sandbox within your project's directory.
4.  **Result Filtering**  
    The sandbox returns *only* the specific filenames and line numbers that match.

</Steps>

## 🛠️ Sandbox API Reference (Advanced)
If you are an AI assistant or a developer writing custom tools, you have access to the `forge` global object:

### `forge.api.searchFiles(pattern, options)`
Perform high-speed `grep` operations locally.
```javascript
// Example: Find all Page Objects missing 'submit' methods
return forge.api.listFiles('pages/')
  .filter(f => !f.content.includes('async submit'));
```

### `forge.api.analyzeCodebase()`
Extract a full map of the repository without reading file contents.
```javascript
// Example: Get a list of every @Given step prefix
const steps = await forge.api.analyzeCodebase();
return steps.map(s => s.regex);
```

### `forge.api.inspectDom(url)`
Returns a pre-filtered, accessibility-scored tree of the target URL.

## 🏗️ Multi-Page Flow Hardening

For E2E flows that span multiple pages, TestForge applies four structural guards that prevent context bloat before it starts.

| Guard | Mechanism | Token Impact |
| :--- | :--- | :--- |
| **No raw ARIA YAML** | `SmartDomExtractor` drops the redundant YAML append | ~1,000 tokens/page saved |
| **Single-page context** | `ContextManager.RECENT_SCANS_TO_KEEP = 1` | Drops prior page DOM on transition |
| **Per-page loop** | `workflow_guide` enforces `inspect → generate → write` per page | Prevents all-DOM accumulation |
| **Relevance filter** | `TestGenerationService` filters steps/POs by keyword match | −40–60% on mature projects |

:::note[Architecture Note]
These guards are **automatic** — no configuration required. They activate when you follow the recommended per-page workflow: inspect one page, generate its test, write to disk, then move to the next.
:::

## 🧠 Knowledge Check: Turbo Mode

<details>
<summary>**Q1: When should I NOT use Turbo Mode (Sandbox)?**</summary>
**Hint**: Sandbox is for analysis, not for modification.
**Answer**: Do not use `execute_sandbox_code` if you actually need to *write* files or *modify* the filesystem. The sandbox is read-only and ephemeral. For writing, use `validate_and_write`.
</details>

<details>
<summary>**Q2: How does the sandbox save 98% in tokens?**</summary>
**Answer**: Traditional agents read a file (1000 tokens), process it in the LLM, and output a result. TestForge runs the processing *locally* in the sandbox. The LLM only receives the 20-token final answer.
</details>
