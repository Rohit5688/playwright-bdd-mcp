## 🏗️ Technical Implementation Patterns

Mastering these patterns ensures your tests are readable by both humans and AI agents.

### 🌐 Handling Complex Iframes
Many enterprise apps (like Stripe or Salesforce) use iframes. TestForge handles these via the `PageController` bridge.

```typescript
// pages/PaymentPage.ts
async fillCardDetails(cardNumber: string) {
    const frame = this.page.frameLocator('iframe[name="card-element"]');
    await frame.getByPlaceholder('Card number').fill(cardNumber);
}
```

### 🚦 Network Interception & Mocking
To test edge cases (like 500 errors), instruct the agent to use the Playwright `route` API within the Page Object:

```typescript
// pages/UserTablePage.ts
async mockEmptyResults() {
    await this.page.route('**/api/users', route => route.fulfill({
        status: 200,
        body: JSON.stringify([])
    }));
}
```

## 🆘 Troubleshooting: The Error Matrix

:::caution[Heads Up]
A `DNA: Locators.Broken` error occurs when the DOM changes. Don't panic!
:::

| Symptom | Probable Root Cause | The Fix 🛠️ |
| :--- | :--- | :--- |
| `DNA: Locators.Broken` | Selector churn in deployment. | Run `heal_and_verify_atomically`. |
| `DNA: Environment.Missing` | Missing `.env` keys. | Use `manage_config` skip sync. |
| `Undefined Step` | Gherkin/Regex mismatch. | Run `analyze_codebase` to re-index. |

## 🧠 Knowledge Check: Failure Recovery

<details>
<summary>**Q1: My test failed with `Assertion.Failed`. Should I run the healer?**</summary>
**Hint**: Assertion failures mean the code worked but the app state was wrong.
**Answer**: No. `Assertion.Failed` usually indicates a real bug in the application logic. Save the logs and report it to the dev team. Only use the healer for `Locators.Broken`.
</details>

<details>
<summary>**Q2: Does 'Atomic Healing' require me to manually copy-paste the fix?**</summary>
**Answer**: No. If you use `heal_and_verify_atomically`, the tool will automatically update your source code file on disk once the fix is verified in the live browser.
</details>

## 🏗️ Knowledge Check: Enterprise Patterns

<details>
<summary>**Q1: Why does TestForge insist on putting locators inside Page classes?**</summary>
**Pro Comment**: This is the **Single Point of Truth** principle. If a button changes, you only fix it in one file (the Page Object), and every test using that button is instantly healed. It prevents 'Locator Rot'.
</details>

<details>
<summary>**Q2: What is the benefit of using `validate_and_write` instead of just writing the file?**</summary>
**Answer**: `validate_and_write` is a safety gate. It runs a background TypeScript compilation to ensure that your new Page Object properly extends `BasePage` and doesn't have any typos that would break your CI/CD pipeline.
</details>
