# TestForge Bug Report: Conflicting Instructions for `setPage` Injection

## Issue Summary
There is a direct contradiction between the output of `get_project_contract` and the prompt generated by `generate_gherkin_pom_test_suite`. This contradiction causes the AI to write invalid step definitions that fail at runtime because the Playwright `page` singleton is never initialized.

## The Contradiction

**1. Output from `get_project_contract` (Correct Requirement):**
```json
"setPageRequired": true
```
> "Use this contract to generate correct code without additional file reads. `setPageRequired=true` means the first Given step MUST destructure `{page}` and call `setPage(page)`."

**2. Output from `generate_gherkin_pom_test_suite` (Conflicting Rules):**
The generated 300+ line rulebook explicitly forbids doing what the contract asks:
> **Rule 84:** `getPage()`/`getRequest()` are pre-wired before each scenario by `test-setup/page-setup.ts` — always available.
> **Rule 130:** This is what connects the `autoSetup` fixture (which calls `setPage(page)`) to every step.
> **Rule 134:** NEVER destructure `{ page }` in a step body — the singleton is already set by the fixture.
> **Rule 146:** ❌ WRONG — do not inject page via fixture destructuring: `Given('I navigate to home', async ({ page }) => { ... })`

## Impact
Because the test generation prompt is so large and aggressively formatted (using terms like "❌ WRONG", "NEVER", and "CRITICAL"), the AI assumes the `autoSetup` fixture handles `setPage` implicitly. It ignores the `get_project_contract` instruction, resulting in step definitions like:

```typescript
// AI generates this:
Given('I am on the login page', async ({}) => {
  await loginPage.navigate(); // CRASHES: page is undefined
});

// Instead of this:
Given('I am on the login page', async ({ page }) => {
  setPage(page);
  await loginPage.navigate();
});
```

When the tests run, the Page Object attempts to call `getPage()`, which returns `undefined`, causing fatal `Cannot read properties of undefined` runtime errors (e.g., when calling `getByLabel`).

## Proposed Fix
The `generate_gherkin_pom_test_suite` tool should be dynamically aware of the `setPageRequired` value from the project contract. 
* If `setPageRequired` is `true`, the rules in the generation prompt should be dynamically updated to **require** `{ page }` destructuring and `setPage(page)` in the first step, removing the hardcoded ban on fixture destructuring.

## Additional Friction Points Encountered
During the scaffolding and testing process, the following minor friction points were also observed. These could be smoothed out for a better out-of-the-box experience:

### 1. The Firefox Binary Trap
**Issue:** `setup_project` generates a `playwright.config.ts` configured for `chromium`, `firefox`, and `webkit`. However, the environment does not always have the Firefox binaries downloaded (it skips or fails `npx playwright install firefox`). 
**Impact:** The very first `run_playwright_test` crashes with a missing binary error for Firefox, forcing the AI to manually edit `playwright.config.ts` to remove the Firefox configuration before testing can proceed.
**Suggestion:** Either `setup_project` should reliably run `npx playwright install` for all configured browsers, or the default `playwright.config.ts` should only target `chromium` to ensure an immediate green build.

### 2. Ghost Imports in `BasePage`
**Issue:** The initial scaffolding tool generates a `BasePage.ts` template that attempts to import a `selectOption` method from `vasu-playwright-utils`.
**Impact:** This method does not actually exist in the `vasu-playwright-utils` package, resulting in immediate TypeScript compilation errors (`Module '"vasu-playwright-utils"' has no exported member 'selectOption'`).
**Suggestion:** Update the `BasePage.ts` boilerplate template in TestForge to remove the `selectOption` import and implement it natively using Playwright's `locator.selectOption()`.

### 3. Strict BDD Fixture Rules (`{}`)
**Issue:** The `playwright-bdd` framework strictly requires the empty object destructuring `{}` as the first argument in step definitions if no other fixtures are used (e.g., `async ({}, arg1)`). 
**Impact:** If the AI writes `async () => { ... }` (which feels natural for parameterless steps), `playwright-bdd` fails to map arguments correctly at runtime. While the TestForge prompt *does* mention this rule, it is buried deep within the 300+ line rulebook and is easy for the LLM to miss, causing test timeouts or argument mismatch errors.
**Suggestion:** Highlight the `{}` injection requirement more prominently in the project contract or as a top-level mandatory rule in the code generation prompt.
