# TestForge MCP — Issues, Bugs & Gaps Report

**Date**: 2026-03-31  
**Tester**: QA Automation Team  
**TestForge MCP version**: 2.4.0  
**Purpose**: Share with TestForge developers to triage and fix gaps found during onboarding.

---

## Executive Summary

During initial onboarding of TestForge MCP onto an existing Playwright-BDD project, we found **11 issues** across 8 tools. The majority stem from a single root cause: **tools ignore the configured project layout in `mcp-config.json` and fall back to hardcoded default scaffold paths**. As a result, TestForge was unable to detect any existing feature files, step definitions, or page objects — even though the project has 60+ files across all three categories — and would have generated code that conflicts with the team's established conventions.

The project uses a custom test framework wrapper package which also could not be resolved, compounding the impact.

### Issue 1 — Custom Wrapper Package Cannot Be Resolved

**Tool**: `analyze_codebase`  
**Severity**: 🔴 High  
**Type**: Bug — Package Resolution Failure

**What happened**:  
The tool was told (via `mcp-config.json → customWrapperPackage`) that the project uses a custom Playwright wrapper installed in the project's `node_modules`. It returned a resolution error and populated `detectedMethods: []` — meaning zero wrapper methods were discovered.

**Why it failed**:  
The tool resolves the package from the MCP server's own working directory, not from the project root. Any package installed only inside the project (especially scoped packages from private registries) will never be found this way.

**Impact**:  
- The AI generation context has no knowledge of the wrapper's API.
- Generated page objects and step definitions will use raw Playwright APIs instead of the team's wrapper methods, producing code that is incompatible with the existing codebase.
- The team will have to manually rewrite every generated file.

**Fix recommendation**:
- Resolve packages using `require.resolve(packageName, { paths: [projectRoot] })` so that the lookup starts from the project being analysed.
- As a fallback, scan `<projectRoot>/node_modules/<packageName>/` for TypeScript declaration files (`.d.ts`) and extract exported method signatures statically — without needing runtime resolution.
- Populate `detectedMethods` from those declarations so the AI generation context is accurate.

### Issue 2 — All Existing Test Files Missed (Features, Steps, Page Objects)

**Tool**: `analyze_codebase`  
**Severity**: 🔴 High  
**Type**: Bug — Scanner ignores configured directories

**What happened**:  
`analyze_codebase` returned empty arrays for `existingFeatures`, `existingStepDefinitions`, and `existingPageObjects`.

**Actual state of the project**:
- **21 feature files** organised across multiple test domains
- **22 step definition files** covering API, authentication, UI, and integration flows
- **18 page object classes** covering every major application screen

**Why it failed**:  
The tool reads `mcp-config.json` and acknowledges the configured directory layout, but the internal scanner still looks in hardcoded default directories (`features/`, `step-definitions/`, `pages/`) rather than using the paths from `mcp-config.json → dirs.*`. None of those default directories exist in this project.

**Impact**:  
- The AI has no knowledge of any existing step patterns — it will create duplicate steps for common flows (login, navigation, assertions) that already exist.
- The AI has no knowledge of existing page objects — it will regenerate page classes, often with a different class pattern than the team uses.
- Downstream tools (`generate_gherkin_pom_test_suite`, `suggest_refactorings`, `validate_and_write`) all receive empty context, producing output that ignores months of prior work.

**Fix recommendation**:
- After loading `mcp-config.json`, use `mcpConfig.dirs.features`, `mcpConfig.dirs.stepDefinitions`, and `mcpConfig.dirs.pages` as the base scan paths.
- These configured values must **fully replace** the hardcoded defaults.
- If a configured path exists but no files are found there, emit a clear warning (e.g., `"Configured features dir 'src/test/features' is empty or not found"`), rather than silently returning an empty array.

### Issue 3 — Environment Config Reporting is Inaccurate

**Tool**: `analyze_codebase`  
**Severity**: 🟠 Medium  
**Type**: Bug — Incorrect env file detection

**What happened**:  
The tool returned a list of `.env` file names in `envConfig.files` but with `keys: []` (empty). The listed file names did not match the actual env files in the project.

**Why it failed**:  
The tool appears to return a pre-determined list of common env file names (`.env`, `.env.integration`, `.env.staging`) rather than scanning the filesystem. It also does not parse the env files to extract key names.

In contrast, the `manage_env` tool (which was tested separately) correctly found the actual `.env` file and reported its keys.

**Impact**:  
- The AI generation context receives incorrect env key information.
- Generated code may reference env variable names that don't exist, or miss keys that do.

**Fix recommendation**:
- `analyze_codebase` should scan the project root and any env-related subdirectories specified in `mcp-config.json` for `.env*` files.
- Only list files that actually exist on disk.
- Parse each found file to populate `keys`.
- Reuse the same detection logic used successfully by `manage_env`.

### Issue 4 — `summarize_suite` Hardcodes Default Path, Ignores Configuration

**Tool**: `summarize_suite`  
**Severity**: 🔴 High  
**Type**: Bug — Ignores `mcp-config.json` dirs

**What happened**:
```
"No features/ directory found. Run setup_project first."
```

**Why it failed**:  
The tool looks for a `features/` directory at the project root, hardcoded. It does not read `mcp-config.json → dirs.features` even though the config file is present and specifies where features live.

**Impact**:  
- The tool is completely non-functional on any project that stores feature files in a non-default location.
- Stakeholder reports and coverage reviews cannot be generated.
- The error message instructs the user to run `setup_project`, which would overwrite their existing test structure.

**Fix recommendation**:
- Read `mcp-config.json → dirs.features` before scanning.
- Use the configured path in all directory lookups.
- Only fall back to `features/` if `mcp-config.json` is absent.
- Surface a clearer error: `"No feature files found at configured path '<configured path>'. Check mcp-config.json → dirs.features."`.

### Issue 5 — `manage_users` Demands Scaffold Even When User Data Already Exists

**Tool**: `manage_users`  
**Severity**: 🟠 Medium  
**Type**: Gap — No detection of existing user stores

**What happened**:
```
"No user store found for environment 'integration'. Run 'scaffold' first."
```

**Why it failed**:  
The tool only recognises user data stored in its own scaffold format (`test-data/users.<env>.json`). It does not check `mcp-config.json → authStrategy`, which explicitly points to the existing user fixture file.

**Actual user data**:  
The project has a rich user fixture file with **150+ named test user personas** organised by feature area, each with environment-specific usernames (integration, staging, dev) and passwords. The user store format is domain-specific — each user has environment-keyed username fields (`dev`, `int`, `stg`) and a password field, rather than a flat single credential set.

**Impact**:  
- Generated code would reference a generic `getUser('admin')` helper pointing to a scaffold file that doesn't contain any of the project's real test users.
- All 150+ user personas would be invisible to TestForge-generated tests.
- Login tests generated by the AI would fail immediately because the user data structure does not match.

**Fix recommendation**:
- Before returning an error, check `mcp-config.json → authStrategy` for an existing user store path.
- If found, read and display its contents rather than requiring a scaffold.
- The `list` action should accept any JSON file path and report its top-level user keys.
- Optionally: support a `userStoreFormat` field in `mcp-config.json` to describe the custom schema so generated code can access users correctly.

### Issue 6 — `validate_and_write` Blocks All File Writes in Non-Default Project Layouts

**Tool**: `validate_and_write`  
**Severity**: 🔴 High  
**Type**: Bug — Security allowlist is hardcoded and overly restrictive

**What happened**:  
Every file path submitted for writing was blocked:
```
⛔ BLOCKED: writes are only allowed in: features, step-definitions, pages, test-data, fixtures, models.
```

**Why it failed**:  
The path allowlist checks whether file paths begin with one of the hardcoded default scaffold directories. It does not consult `mcp-config.json → dirs.*` to determine what the valid write locations are for *this* project.

Since every file was at a non-default path (matching the project's configured layout), every write was blocked.

**Impact**:  
- **This is a complete blocker.** No generated file can be written to disk for any project using a non-default directory layout.
- The generation workflow is entirely non-functional on this and likely many other real-world projects.

**Fix recommendation**:
- Load `mcp-config.json → dirs` before applying the allowlist.
- Build the allowed prefixes dynamically:
  ```
  allowedPrefixes = [dirs.features, dirs.pages, dirs.stepDefinitions, dirs.testData]
  ```
- Only fall back to the hardcoded list if `mcp-config.json` is absent or `dirs` is not configured.

### Issue 7 — `self_heal_test` Cannot Classify Common Playwright Failures

**Tool**: `self_heal_test`  
**Severity**: 🟠 Medium  
**Type**: Bug — Incomplete failure classifier

**What happened**:  
A standard Playwright error (`locator resolved to hidden element`) was submitted. The tool returned:
```
[SELF-HEALING: UNKNOWN FAILURE]
The failure could not be automatically classified.
```

**Why it failed**:  
The classifier matches a narrow set of error message patterns (e.g., `"waiting for"`, `"strict mode violation"`). The phrase `"resolved to hidden element"` is one of the most common Playwright locator errors but is not in the pattern list.

**Impact**:  
- Self-healing provides no value for one of the most frequent failure types.
- Engineers receive no fix suggestion and must diagnose manually.

**Fix recommendation**:  
Extend the pattern matcher to cover common Playwright failure strings:

| Error string | Classification | Suggested fix |
|---|---|---|
| `"resolved to hidden element"` | `LOCATOR_HIDDEN` | Wait for visibility or use `{ force: true }` |
| `"element is outside of the viewport"` | `SCROLL_ISSUE` | Scroll element into view before acting |
| `"intercepts pointer events"` | `OVERLAY_BLOCKING` | Dismiss overlay/modal before acting |
| `"waiting for element to be visible"` | `WAIT_TIMEOUT` | Increase timeout or check selector |
| `"locator.click: Timeout"` | `CLICK_TIMEOUT` | Verify element exists and is interactive |

### Issue 8 — `suggest_refactorings` Returns False-Positive "Clean" Result

**Tool**: `suggest_refactorings`  
**Severity**: 🟠 Medium  
**Type**: Bug — Scanner uses wrong paths

**What happened**:
```
✅ No unused Page Object methods detected.
✅ No duplicate step definition patterns detected.
🎉 Your codebase is incredibly clean!
```

**Why it failed**:  
Same root cause as Issue 2: the scanner looks in default directories (`step-definitions/`, `pages/`). These directories don't exist. The scanner finds zero files and reports zero issues — a false positive.

The actual project has 22 step files and 18 page objects. In a codebase of this size, finding zero duplicates or unused methods is not credible.

**Impact**:  
- Engineers trust the report and miss genuine refactoring opportunities.
- The tool provides a false sense of code quality.

**Fix recommendation**:
- Resolve dirs from `mcp-config.json` before scanning, same fix as Issue 2.
- When no files are found at the configured paths, return a warning: `"No files found at configured paths — refactoring analysis skipped."` rather than a false "clean" result.

### Issue 9 — `generate_gherkin_pom_test_suite` Returns Prompt Text Instead of Generated Files

**Tool**: `generate_gherkin_pom_test_suite`  
**Severity**: 🔴 High  
**Type**: Design Gap — Tool is a prompt builder, not a generator

**What happened**:  
Calling the tool returned several kilobytes of structured instruction text (a system prompt), not the feature file, step definitions, and page object that were requested.

**Why this is a design gap**:  
The tool is designed to construct a generation prompt for the *calling AI host* to act on. The tool itself does not generate any code. This means:
- In an automated or scripted pipeline, the output is unusable — no files are produced.
- The tool name (`generate_gherkin_pom_test_suite`) strongly implies it produces a test suite; it should be documented clearly that it requires an AI host to complete the generation step.
- There is no defined contract between the prompt output and the `validate_and_write` input — the calling AI must bridge this gap correctly every time.

**Impact**:  
- The tool cannot be integrated into a CI/CD pipeline without an AI intermediary.
- Different AI hosts may interpret the prompt differently, producing inconsistent output.

**Fix recommendation**:
- **Option A (preferred)**: Integrate an LLM call inside the tool. Accept an API key config and call the generation model directly, returning a structured JSON of files ready to pass to `validate_and_write`.
- **Option B**: Clearly rename the tool (e.g., `build_generation_context`) and document that the caller must pass its output to an LLM and then to `validate_and_write`.
- **Option C**: Add a `generateFiles: true` parameter. When set, the tool pipes the prompt through an internal model and returns the generated file set.

### Issue 10 — `playwrightConfig` Field in `mcp-config.json` Is Unused

**Tool**: All tools that invoke Playwright  
**Severity**: 🟡 Low  
**Type**: Gap — Config field not consumed

**What happened**:  
The `playwrightConfig` field pointing to the non-default Playwright config file was stored in `mcp-config.json` and read back correctly by `manage_config`. However, no other tool (including `run_playwright_test`) reads or uses this field. Tools that run Playwright commands appear to use a hardcoded default config path.

**Impact**:  
- Test runs invoked by TestForge would use the wrong config file, missing environment setup, BDD configuration, reporters, and custom timeouts.

**Fix recommendation**:
- Any tool that invokes `npx playwright test` or `npx bddgen` should read `mcp-config.json → playwrightConfig` and pass `--config=<value>` to the command.
- Document `playwrightConfig` as a first-class config field in the `manage_config` scaffold template.

### Issue 11 — BDD Setup Incorrectly Reported as Absent

**Tool**: `analyze_codebase`  
**Severity**: 🟠 Medium  
**Type**: Bug — BDD detection uses wrong file path

**What happened**:
```json
"bddSetup": { "present": false },
"recommendation": "Playwright-BDD config not found. Standard setup needed."
```

**Why it failed**:  
The BDD detection logic scans only `playwright.config.ts` at the project root. The project's Playwright config is located at a non-default path (as recorded in `mcp-config.json → playwrightConfig`). The root-level config had been removed by the user, so detection returned false.

**Actual BDD state**: Fully configured — `playwright-bdd` is imported, `defineBddConfig` is called, and two BDD-related npm scripts exist in `package.json` (`generate:features` using `bddgen`, `automated-test:bdd` using `--config` pointing to the correct file).

**Impact**:  
- The tool recommends running `setup_project`, which would scaffold an entirely new BDD structure and risk overwriting the existing configuration.
- All generation context assumes BDD is not present and omits reuse of existing step patterns.

**Fix recommendation**:
- Read `mcp-config.json → playwrightConfig` and scan *that* file for `playwright-bdd` imports and `defineBddConfig` calls.
- Use `package.json → scripts` as a secondary BDD signal: if any script references `bddgen`, mark `bddSetup.present: true`.
- Never return `false` without checking both signals.

## Cross-Cutting Root Cause

Nearly all issues (2, 4, 6, 8, 11) share a single architectural flaw:

> **Tools are hardcoded to a default scaffold layout and do not use `mcp-config.json → dirs` as the source of truth for project paths.**

The config file exists precisely to describe non-standard layouts. If tools ignore it, the config is pointless and the tool only works on freshly scaffolded projects.

**Required fix across all affected tools**:

| Config field | Default (currently used) | Must use when configured |
|---|---|---|
| `dirs.features` | `features/` | value from config |
| `dirs.pages` | `pages/` | value from config |
| `dirs.stepDefinitions` | `step-definitions/` | value from config |
| `dirs.testData` | `test-data/` | value from config |
| `playwrightConfig` | `playwright.config.ts` | value from config |

## Developer Implementation Guide

This section goes beyond "what to fix" to explain **how the tool should behave for any project** — not just this one. It is intended to give the developer a complete mental model of the expected behaviour before writing a single line of code.

---

### Core Principle: `mcp-config.json` Is the Contract

The fundamental contract TestForge offers to teams is:

> "Configure your project layout once in `mcp-config.json`, and every tool will respect it."

Currently that contract is broken. Every service reads `mcp-config.json` via `McpConfigService` correctly, but then **ignores the `dirs.*` values it just read** and uses hardcoded strings instead. This is the single most impactful thing to fix.

The correct pattern — which should be applied uniformly across all services — is:

```typescript
// ✅ CORRECT: resolve directories from config
const config = mcpConfig.read(projectRoot);
const featuresDir = path.join(projectRoot, config.dirs.features);
const stepsDir    = path.join(projectRoot, config.dirs.stepDefinitions);
const pagesDir    = path.join(projectRoot, config.dirs.pages);
const testDataDir = path.join(projectRoot, config.dirs.testData);

// ❌ WRONG: hardcode the scaffold defaults
const featuresDir = path.join(projectRoot, 'features');
const stepsDir    = path.join(projectRoot, 'step-definitions');
const pagesDir    = path.join(projectRoot, 'pages');
```

**Every service that reads or writes files must follow this pattern.** There should be zero hardcoded directory strings in any service other than `McpConfigService` itself (where they define the defaults).

---

### Where Each Fix Lives in `index.ts`

Reading `index.ts` reveals that several bugs are most efficiently fixed at the **tool handler level** rather than inside individual services. This is the place where `McpConfigService` is already called and where the resolved config is available to pass down.

---

#### `validate_and_write` handler (line ~310 in `index.ts`)

**Current code**:
```typescript
case "validate_and_write": {
  const { projectRoot, files, pageUrl, dryRun } = args as any;
  // ...
  const writeResult = fileWriter.writeFiles(projectRoot, files, false);
  //                                        ^^^^^^^^^^^^^^^^^^^^^^^^^^^
  //                  config dirs are NEVER passed — allowlist is always hardcoded defaults
```

**Fix**: Read config before calling `writeFiles()` and pass the configured dirs:
```typescript
case "validate_and_write": {
  const { projectRoot, files, pageUrl, dryRun } = args as any;
  const config = mcpConfig.read(projectRoot);
  const configuredDirs = [
    config.dirs.features,
    config.dirs.pages,
    config.dirs.stepDefinitions,
    config.dirs.testData,
  ];
  const writeResult = fileWriter.writeFiles(projectRoot, files, dryRun ?? false, configuredDirs);
  // ...
```

This is the minimal change needed to unblock all file writes for any non-default project layout. No changes needed inside `FileWriterService` except adding the `extraAllowedDirs` parameter.

#### `run_playwright_test` handler (line ~250 in `index.ts`)

**Current code**:
```typescript
case "run_playwright_test": {
  const { projectRoot, specificTestArgs, tags } = args as any;
  const config = mcpConfig.read(projectRoot);
  const grepArg = tags ? `--grep "${tags}"` : '';
  const combinedArgs = [specificTestArgs, grepArg].filter(Boolean).join(' ');
  const result = await runner.runTests(projectRoot, combinedArgs || undefined, config.testRunTimeout);
  //                              Note: no --config flag passed ^
```

`config.testRunTimeout` is read but `config.playwrightConfig` is ignored. The `runner.runTests()` call constructs a command like `npx bddgen && npx playwright test`, using the project root's default `playwright.config.ts`.

**Fix**: Pass `--config` when `playwrightConfig` is set:
```typescript
case "run_playwright_test": {
  const { projectRoot, specificTestArgs, tags } = args as any;
  const config = mcpConfig.read(projectRoot);
  const configFlag = (config as any).playwrightConfig
    ? `--config=${(config as any).playwrightConfig}`
    : '';
  const grepArg = tags ? `--grep "${tags}"` : '';
  const combinedArgs = [configFlag, specificTestArgs, grepArg].filter(Boolean).join(' ');
  const result = await runner.runTests(projectRoot, combinedArgs || undefined, config.testRunTimeout);
```

The same fix applies to `update_visual_baselines` and the `runner.runTests()` call inside `validate_and_write`.

#### `manage_users list` handler (line ~390 in `index.ts`)

**Current code**:
```typescript
if (action === "list") {
  const storeResult = userStore.read(projectRoot, env);
  if (!storeResult.exists) {
    return { content: [{ type: "text", text: `No user store found for environment "${env}". Run 'scaffold' first.` }] };
  }
```

The `userStore.read()` call goes directly to `test-data/users.{env}.json`. It never checks `mcp-config.json → authStrategy`.

**Fix in `UserStoreService.read()`**: Before checking the default path, inspect `authStrategy`:
```typescript
public read(projectRoot: string, environment: string): UserStoreReadResult {
  const config = new McpConfigService().read(projectRoot);

  // If authStrategy is a file path (not a keyword), try reading it directly
  const kwds = ['none', 'users-json', 'env'];
  if (config.authStrategy && !kwds.includes(config.authStrategy)) {
    const resolved = path.isAbsolute(config.authStrategy)
      ? config.authStrategy
      : path.join(projectRoot, config.authStrategy);
    if (fs.existsSync(resolved)) {
      try {
        const raw = JSON.parse(fs.readFileSync(resolved, 'utf-8'));
        const roles = Object.keys(raw).filter(k => !k.startsWith('_') && typeof raw[k] === 'object');
        return { environment, filePath: resolved, exists: true, roles, users: raw };
      } catch {}
    }
  }

  // original logic ...
  const testDataDir = path.join(projectRoot, this.TEST_DATA_DIR);
  const filePath = this.storeFile(testDataDir, environment);
  // ...
}
```

With this change, `manage_users list` works on any project regardless of where user data is stored.

### `McpConfig` Interface — Missing `playwrightConfig` Field

`McpConfigService.ts` defines the `McpConfig` interface. The `playwrightConfig` field is referenced throughout the project (`mcp-config.json` files set it, `manage_config` returns it) but it is **absent from the TypeScript interface**. This means:

1. TypeScript will not catch typos when accessing `config.playwrightConfig`.
2. The `scaffold` function never writes it to new `mcp-config.json` files, so teams that use `setup_project` never get the field even if they need it.
3. The `DEFAULT_CONFIG` has no default, so `(config as any).playwrightConfig` type-casts are required everywhere — a clear sign the field should be formalised.

**Fix** — add to `McpConfig` interface and `DEFAULT_CONFIG`:
```typescript
// In McpConfig interface:
/** Relative or absolute path to the Playwright config file. Defaults to 'playwright.config.ts'. */
playwrightConfig: string;

// In DEFAULT_CONFIG:
playwrightConfig: 'playwright.config.ts',
```

Once this is done, all `(config as any).playwrightConfig` casts can be removed and the field is available without casting throughout the codebase.
